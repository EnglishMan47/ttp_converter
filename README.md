# Информационная система пакетной обработки отсканированных документов (веб-версия)

Преобразует сканы книг (TIFF) в поисковые PDF/A: изображение страницы +
невидимый текстовый слой, закладки по заголовкам, метаданные документа.
Работает как **веб-приложение**: сервер запускается Docker-контейнером,
пользователи работают через браузер без установки чего-либо у себя.

## Возможности

- Клиент-серверная модель: загрузка сканов и скачивание результатов
  через браузер
- Два движка распознавания, оба готовы сразу после первого старта
  контейнера:
  - **лёгкий** (EasyOCR) — быстрый, работает на CPU, подходит для
    любого сервера;
  - **нейросетевой** — квантованная модель Chandra Q4_K_M (~3 ГБ вместо
    10 ГБ у полной, качество почти не отличается): понимает структуру страницы
    (заголовки, таблицы). На видеокарте NVIDIA (6+ ГБ) быстрый, без
    неё работает на процессоре (медленно — минуты на страницу)
- Двухфазный запуск: добавленная книга не обрабатывается, пока
  пользователь явно не нажмёт «Начать» — случайный запуск исключён
- Индивидуальные настройки каждой книги: разрешение (DPI) и число
  страниц на проход; общие настройки партии — движок и предобработка
- Строгая очередь: книги обрабатываются в порядке запуска; остановка
  или ошибка одной книги не блокирует следующие; повторный запуск
  остановленной книги встаёт в конец очереди и продолжается с
  контрольной точки
- Карточки завершённых/остановленных книг можно закрывать
- Время на карточке книги: сколько прошло, сколько приблизительно
  осталось, в какое время обработка завершилась; над списком — общая
  статистика партии (прошло / осталось / когда завершена)
- Результаты по книге: `<имя>.pdf` (PDF/A), `<имя>_review.pdf`
  (подсветка сомнительных слов), `<имя>_report.txt` (отчёт)

## Установка (Docker — основной способ)

Требования: 64-разрядная система (Linux, Windows 10/11 или macOS),
15+ ГБ свободного диска, доступ в интернет при первом старте. Память:
для лёгкого движка достаточно 4+ ГБ, для нейросетевого (Chandra на CPU)
нужно **10–12 ГБ ОЗУ**.
На Windows/macOS контейнеру достаётся лишь часть ОЗУ хоста: если
нейросетевой движок сообщает о нехватке памяти, увеличьте лимит в
Docker Desktop → Settings → Resources → Memory до 10–12 ГБ.

### Шаг 1. Установите Docker

**Debian / Ubuntu:**

```bash
sudo apt update && sudo apt install -y git curl
curl -fsSL https://get.docker.com | sudo sh
sudo usermod -aG docker $USER
```

После `usermod` выйдите из сеанса и войдите снова (или выполните
`newgrp docker`), иначе понадобится sudo перед каждой docker-командой.

**Windows / macOS:** установите
[Docker Desktop](https://www.docker.com/products/docker-desktop/) и
дождитесь статуса Running; git — с
[git-scm.com](https://git-scm.com/download/win).

### Шаг 2. Скачайте проект

```bash
git clone https://github.com/EnglishMan47/ttp_converter.git
cd ttp_converter
```

### Шаг 3. Проверьте готовность системы

Скрипт проверяет docker и compose, запущенный движок, архитектуру,
свободный диск, память, занятость порта 8501 и доступ к серверам
загрузки; ничего не устанавливает.

Linux/macOS (терминал):

```bash
bash scripts/preflight.sh
```

Windows (командная строка):

```bat
scripts\preflight.bat
```

Устраните строки `[ошибка]`, на `[важно]` — обратите внимание.

### Шаг 4. Соберите и запустите контейнер

```bash
docker compose up -d --build
docker compose logs -f        
```

Лёгкий движок (EasyOCR) работает **сразу** — его модели встроены в
проект (`app/resources/easyocr`) и приезжают вместе с клоном, ничего не
скачивая. Для нейросетевого движка при **первом** старте контейнер
догружает две вещи (одноразово, сохраняются в томах Docker):

| Что | Объём | Куда |
|---|---|---|
| Runtime нейросетевого движка (llama.cpp) | ~50 МБ | том `neural` |
| Веса модели Chandra Q4_K_M + проектор | ~3.7 ГБ | том `models-cache` |

**Веб-интерфейс доступен сразу** — не нужно ждать окончания загрузок.
Пока модели скачиваются, страница http://localhost:8501 показывает
статус подготовки и обновляется сама; когда всё готово, открывается
обычный интерфейс. Ту же готовность видно и в логах:

```
======================================================================
  Всё готово! Откройте веб-интерфейс в браузере:

      http://localhost:8501           — с этого компьютера
      http://<адрес-сервера>:8501     — с других компьютеров сети
======================================================================
```

### Видеокарта NVIDIA (необязательно)

С видеокартой NVIDIA (от 6 ГБ видеопамяти) нейросетевой движок работает
в разы быстрее, чем на процессоре. **По умолчанию видеокарта не
используется** — её нужно включить явно (шаги ниже), иначе контейнер её
просто не увидит и будет считать на процессоре.

Сразу о CUDA, чтобы не путаться — она нужна в двух разных местах:

| Что | Где | Кто ставит |
|---|---|---|
| **Драйвер NVIDIA** (даёт `nvidia-smi`) | на хосте | **вы, вручную** — шаг 1 |
| **Проброс видеокарты в контейнер** | на хосте | Linux — вручную (`nvidia-container-toolkit`); **Windows — сам Docker Desktop** |
| **CUDA-toolkit** (компилятор `nvcc` для сборки движка) | внутри контейнера | **ставится сам** при первом старте |

То есть **сам CUDA-toolkit устанавливать не нужно** — контейнер при
первом старте сам поставит его внутри себя (`nvidia-cuda-toolkit`,
CUDA 12.0 из репозитория Ubuntu, ~75 пакетов) и соберёт llama.cpp с
`-DGGML_CUDA=ON`.

Шаг 1 зависит от вашей системы — выберите **один** из двух вариантов
ниже. Команды с `sudo`/`apt` относятся **только к Linux-серверу**: на
Windows их выполнять не нужно и негде.

#### Шаг 1А. Если хост — Linux-сервер

Драйвер (проверить/поставить):

```bash
nvidia-smi                        # есть таблица с видеокартой — драйвер стоит
sudo ubuntu-drivers autoinstall   # если нет: Ubuntu подберёт версию сама
sudo reboot
```

Затем `nvidia-container-toolkit` — он пробрасывает видеокарту в
контейнеры (команды из официальной инструкции NVIDIA):

```bash
# репозиторий и ключ NVIDIA
curl -fsSL https://nvidia.github.io/libnvidia-container/gpgkey \
  | sudo gpg --dearmor -o /usr/share/keyrings/nvidia-container-toolkit-keyring.gpg \
  && curl -s -L https://nvidia.github.io/libnvidia-container/stable/deb/nvidia-container-toolkit.list \
  | sed 's#deb https://#deb [signed-by=/usr/share/keyrings/nvidia-container-toolkit-keyring.gpg] https://#g' \
  | sudo tee /etc/apt/sources.list.d/nvidia-container-toolkit.list

# установка
sudo apt-get update
sudo apt-get install -y nvidia-container-toolkit

# привязка к Docker и перезапуск
sudo nvidia-ctk runtime configure --runtime=docker
sudo systemctl restart docker
```

#### Шаг 1Б. Если хост — Windows 10/11 с Docker Desktop

1. **Драйвер NVIDIA для Windows** — свежий, с поддержкой WSL 2 GPU
   Paravirtualization (обычный драйвер с сайта NVIDIA; ставится в
   Windows, а не внутри WSL).
2. **Свежее ядро WSL 2** — в обычной командной строке:

   ```bat
   wsl --update
   ```

3. **Бэкенд WSL 2 в Docker Desktop** — Settings → General → галочка
   «Use the WSL 2 based engine» (включена по умолчанию).

#### Проверка, что Docker видит видеокарту (обе системы)

```bash
docker run --rm --gpus all ubuntu:24.04 nvidia-smi
```

Должна появиться таблица с вашей видеокартой. Если да — переходите к
шагу 2.

#### Шаг 2. Включите видеокарту в проекте

Откройте **`docker-compose.yml`** в корне папки проекта любым текстовым
редактором и найдите эти семь строк:

```yaml
    # deploy:
    #   resources:
    #     reservations:
    #       devices:
    #         - driver: nvidia
    #           count: all
    #           capabilities: [gpu]
```

В начале каждой из семи строк удалите два символа — решётку и пробел за
ней (`# `). Должно получиться:

```yaml
    deploy:
      resources:
        reservations:
          devices:
            - driver: nvidia
              count: all
              capabilities: [gpu]
```

Сохраните файл и перезапустите контейнер:

```bash
docker compose up -d --build
docker compose logs -f
```

#### Что произойдёт при первом старте с видеокартой

Контейнер сам, без вашего участия:

1. установит внутри себя `nvidia-cuda-toolkit` (~75 пакетов, несколько ГБ);
2. скачает исходники llama.cpp и соберёт её с `-DGGML_CUDA=ON` под вашу
   видеокарту (`CMAKE_CUDA_ARCHITECTURES=native`).

Это разово и **занимает десятки минут** — результат сохраняется в томе
`neural`, при следующих стартах берётся готовым. Веб-интерфейс при этом
доступен сразу и показывает статус подготовки.

Если CUDA-сборка почему-то не удастся, контейнер **не останется без
движка**: он поставит готовую CPU-сборку и напишет в логах
«CUDA-сборка не удалась… движок будет работать на процессоре».

#### Проверка

```bash
docker compose exec tiff2pdf nvidia-smi   # контейнер видит видеокарту?
docker compose logs | grep neural         # что именно установилось
```

В логах при успешной GPU-сборке будет «Сборка из исходников с CUDA» и
затем «Сборка завершена (cuda)».

#### Если что-то пошло не так

- **В логах «Видеокарта NVIDIA не обнаружена», хотя она есть** — блок
  `deploy` не раскомментирован (шаг 2); на Linux — ещё не установлен
  `nvidia-container-toolkit`, на Windows — старый драйвер или не
  обновлено ядро WSL 2 (`wsl --update`), см. шаг 1.
- **Собралась CPU-сборка вместо CUDA** — смотрите логи. Частые причины:
  недоступен `github.com` (исходники тянутся оттуда; в сети без VPN
  сборка невозможна) либо очень новая видеокарта: Ubuntu 24.04 даёт
  CUDA **12.0**, а картам поколения RTX 50xx (Blackwell) нужна CUDA 12.8+
  — тогда `nvcc` не знает их архитектуру. В этом случае нужен более
  свежий CUDA-toolkit в образе (замена базового образа на
  `nvidia/cuda:12.8-devel-ubuntu24.04` в `Dockerfile`) — напишите, если
  столкнётесь.

## Управление контейнером

Все команды выполняются из папки проекта.

| Действие | Команда |
|---|---|
| Посмотреть логи | `docker compose logs -f` (или `docker logs -f tiff2pdf`) |
| Статус | `docker compose ps` |
| Остановить | `docker compose stop` |
| Запустить снова | `docker compose start` |
| Перезапустить | `docker compose restart` |
| Остановить и удалить контейнер (тома с моделями сохраняются) | `docker compose down` |
| Запустить после `down` | `docker compose up -d` |
| Обновить до свежей версии | `git pull && docker compose up -d --build` |

Контейнер имеет фиксированное имя `tiff2pdf`, поэтому к нему можно
обращаться напрямую — например `docker logs tiff2pdf` или
`docker exec -it tiff2pdf bash`.

Контейнер запускается с политикой `restart: unless-stopped` — после
перезагрузки сервера он поднимется сам (если запущена служба Docker).

**Полное удаление** — контейнер, тома со скачанными моделями, образ и
папка проекта.

Linux/macOS (терминал):

```bash
docker compose down -v --rmi local   # контейнер + тома + образ
docker builder prune -f              # кэш сборки (по желанию)
cd .. && rm -rf ttp_converter        # папка проекта
```

Windows (командная строка):

```bat
docker compose down -v --rmi local
docker builder prune -f
cd .. && rd /s /q ttp_converter
```

После этого от программы в системе не остаётся ничего, кроме самого
Docker.

## Пример результата (`example_test/`)

В папке [example_test](example_test) — готовый пример обработки:

- `uploads/book1/` — исходные TIFF-сканы (страницы дореволюционных
  книг);
- `output/EasyOCR/` — результат лёгкого движка: `book1.pdf`
  (поисковый PDF/A), `book1_review.pdf` (подсветка сомнительных слов),
  `book1_report.txt` (отчёт);
- `output/Chandra/` — та же книга, обработанная нейросетевым движком.

Откройте оба PDF, попробуйте поиск по тексту и сравните качество
слоёв. Эти же TIFF-файлы удобно использовать для первой проверки
собственной установки.

## Работа пользователя

1. Открыть адрес сервера в браузере.
2. Загрузить книгу: ZIP с TIFF-страницами (имя архива = название книги)
   или отдельные TIFF-файлы.
3. При необходимости поправить DPI и «страниц на проход» у книги.
4. Выбрать движок в боковой панели («Авто» сам подберёт подходящий).
5. Нажать «Начать» у книги или «Начать обработку всех».
6. Следить за прогрессом трёх этапов; книгу можно остановить
   («Стоп»/«Остановить всё») и позже продолжить с места остановки.
7. Скачать результаты кнопками на карточке готовой книги.
8. Неактуальные карточки закрывать кнопкой «Закрыть» (готовые PDF при
   этом не удаляются до перезапуска сервера — скачайте их заранее).

## Установка без Docker (альтернатива)

```bash
bash setup.sh          # Linux;  setup.bat — Windows
source venv/bin/activate
streamlit run app/web.py
```

Скрипт перед установкой проверяет git, версию Python (нужна 3.10+),
модуль venv, свободное место на диске и сообщает, доступен ли
Docker-вариант. Быстрая самопроверка без установки: `./setup.sh check`
(Linux) или `setup.bat check` (Windows). Сразу после установки доступен
лёгкий движок (EasyOCR); нейросетевой добавляется отдельно (ниже).

### Нейросетевой движок на Windows без Docker

Windows (командная строка, из папки проекта):

```bat
scripts\setup_neural_windows.bat
```

Скрипт скачивает официальную CPU-сборку llama.cpp (утилита
`llama-mtmd-cli`, ~20 МБ) в папку `neural\llama.cpp` проекта —
приложение находит её автоматически, в боковой панели появится
«Нейросетевой движок: установлен». Веса модели Chandra Q4_K_M (~3.7 ГБ)
скачиваются в `neural\models` при первом использовании движка либо
сразу — командой `scripts\setup_neural_windows.bat -DownloadModels`.

Если на системном диске мало места, ДО установки перенаправьте папку
`neural` на другой диск (junction, права администратора не нужны):

```
mkdir D:\tiff2pdf_neural
mklink /J "<папка проекта>\neural" D:\tiff2pdf_neural
```

Без видеокарты модель работает на процессоре — порядка 1–3 минут на
страницу; модель загружается в память заново на каждую страницу.
Большие партии быстрее обрабатывать лёгким движком, нейросетевой
включать выборочно — он даёт структуру страницы (заголовки, таблицы)
и точнее на сложной вёрстке. Путь к собственной сборке llama.cpp
задаётся переменной `LLAMA_MTMD_BIN`, каталог весов —
`CHANDRA_MODELS_DIR`.

## Архитектура

```
app/
├── web.py             веб-интерфейс (Streamlit)
├── queue_manager.py   очередь книг, фоновая обработка, статусы
├── report.py          отчёты по книгам и партии
├── stage_1/           этап 1: предобработка сканов
├── stage_2/           этап 2: распознавание
│   ├── engines.py     выбор движка по оборудованию
│   ├── ocr_easy.py    лёгкий движок (EasyOCR)
│   ├── ocr_chandra.py нейросетевой движок (Chandra GGUF Q4_K_M)
│   └── jsonl_store.py общий формат контрольных точек
├── stage_3/           этап 3: сборка PDF/A с текстовым слоем
└── resources/         шрифт текстового слоя, цветовой профиль
```

Нейросетевой движок: модель `chandra-ocr-2.Q4_K_M.gguf` и проектор
`chandra-ocr-2.mmproj-bf16.gguf` из репозитория
`prithivMLmods/chandra-ocr-2-GGUF` (Hugging Face), инференс через
`llama-mtmd-cli` (llama.cpp). В Docker всё это устанавливается
автоматически при первом старте контейнера; выбор движка на партию —
в боковой панели интерфейса.
